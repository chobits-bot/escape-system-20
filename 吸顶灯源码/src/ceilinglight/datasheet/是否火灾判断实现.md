# 数据融合计算是否火灾 —— 完整设计方案

将 **MLX90640（热成像）**、**ENS160（空气质量/VOC）** 和 **AHT2x（温湿度）** 三者的数据融合，本质上是构建一个 **“感-知-判”一体化的早期火灾预警系统**。火灾发生前，环境中的温度、气体和湿度会呈现特定的变化模式，多传感器融合正是为了捕捉这些细微征兆，实现比单一传感器更早、更准确的预警。

根据学术研究和工程实践，采用 **“特征层提取 → 特征层融合 → 决策层融合”** 的三级架构。整体流程如下：

------

## 一、系统架构总览

```
┌─────────────────┐
│   MLX90640       │── ΔT, T_rise_rate ─┐
└─────────────────┘                     │
┌─────────────────┐                     │
│   ENS160         │── TVOC ────────────┼── [模糊逻辑] ── BPA₁/BPA₂/BPA₃ ── [D-S组合] ── 火灾概率
└─────────────────┘                     │
┌─────────────────┐                     │
│   AHT2x          │── RH_rate ────────┘
└─────────────────┘
```

| 层级           | 功能                           | 算法/方法                          |
| :------------- | :----------------------------- | :--------------------------------- |
| **数据预处理** | 将原始传感器读数转化为物理特征 | 温湿度补偿、滑动窗口统计、线性回归 |
| **特征层融合** | 将特征值映射为证据（BPA）      | 模糊逻辑（梯形隶属度 + 规则推理）  |
| **决策层融合** | 组合多个证据，输出最终判断     | D-S 证据理论（Dempster-Shafer）    |
| **决策输出**   | 多级报警                       | 安全 / 关注 / 预警 / 报警          |

## 第一步：数据预处理与特征提取

在融合之前，需要将原始数据转化为有物理意义的特征，AHT2x 的数据在这一步发挥关键作用。

### 1.1 AHT2x 温湿度补偿

- **补偿 ENS160**：ENS160 的 VOC 和 eCO₂ 读数受温湿度影响显著。将 `Ta` 和 `RH` 写入 ENS160 的补偿寄存器，能确保其输出在不同环境下的一致性。
- **补偿 MLX90640**：MLX90640 的温度计算本身就依赖环境温度 `Ta`（已在传感器驱动层完成）。

### 1.2 提取火灾特征指标

| 传感器       | 原始数据       | 提取特征                      | 物理意义                   | 计算方法                       |
| :----------- | :------------- | :---------------------------- | :------------------------- | :----------------------------- |
| **MLX90640** | 32×24 温度矩阵 | `T_max`：视场最高温度         | 热辐射异常的直接证据       | 取矩阵最大值                   |
|              |                | `ΔT = T_max - Ta`             | 排除环境温度变化的干扰     | 最高温 - 环境温度              |
|              |                | `T_rise_rate`：温度上升速率   | 早期火灾的敏感指标         | 最近 N 帧的线性回归斜率        |
|              |                | `Hot_spot_area`：高温像素面积 | 区分局部高温与噪声干扰     | 统计 > 阈值（如 60°C）的像素数 |
| **ENS160**   | TVOC (ppb)     | `TVOC` 实时浓度               | 材料热解释放的挥发性有机物 | 直接读取                       |
|              | eCO₂ (ppm)     | `eCO₂` 实时浓度               | 明火燃烧释放的二氧化碳     | 直接读取                       |
|              | AQI (0~5)      | 空气质量指数                  | 综合参考指标               | 直接读取                       |
| **AHT2x**    | 湿度 RH (%)    | `RH` 实时值                   | 环境湿度基线               | 直接读取                       |
|              |                | `RH_rate`：湿度变化率         | 火灾蒸发或燃烧产水         | 滑动窗口线性回归斜率           |

### 1.3 特征计算的滑动窗口参数

| 特征            | 窗口大小 | 更新频率 | 说明                     |
| :-------------- | :------- | :------- | :----------------------- |
| `T_rise_rate`   | 5 帧     | 每帧     | 线性回归斜率，单位 °C/帧 |
| `RH_rate`       | 5 帧     | 每帧     | 线性回归斜率，单位 %/帧  |
| `Hot_spot_area` | 1 帧     | 每帧     | 实时统计                 |

## 第二步：特征层融合 —— 模糊逻辑

模糊逻辑概念理解：

> 传统逻辑（布尔逻辑） vs. 模糊逻辑

- **传统逻辑（二值逻辑）**：像开关一样，非黑即白。
  - 问题：室内温度是“高温”吗？
  - 答案：我们设定阈值 50°C。低于 50°C 就是 **“假”**（不是高温），高于 50°C 就是 **“真”**（是高温）。
  - **缺陷**：如果温度是 49.9°C，它被判定为“不是高温”，这显然不符合我们的直觉（49.9°C 已经非常烫了）。
- **模糊逻辑**：像无极调光的旋钮，允许存在“部分真实”。
  - 问题：室内温度是“高温”吗？
  - 答案：不需要死板的阈值。模糊逻辑会给出一个 **“隶属度”**（0 到 1 之间的值）。
  - **表现**：
    - 25°C 时，“高温”的隶属度是 0（完全不烫）。
    - 45°C 时，“高温”的隶属度是 0.7（相当烫了）。
    - 60°C 时，“高温”的隶属度是 1.0（绝对烫）。

模糊逻辑能够很好地处理传感器数据的不确定性和非线性关系，鲁棒性强。有研究明确提出，以温度、烟雾浓度和红外火焰信号为模糊输入量，构建多信息模糊融合系统，能有效降低误判率。

### 2.1 模糊化（隶属度函数）

所有输入变量均采用 **梯形隶属度函数**（计算量小，适合嵌入式实现）。梯形隶属度函数由四个参数 `(a, b, c, d)` 定义，隶属度值在 `[a, b]` 区间上升，在 `[b, c]` 区间保持为 1，在 `[c, d]` 区间下降。

#### 2.1.1 温度风险 `ΔT`（单位：℃）

| 模糊集      | a    | b    | c    | d    | 物理含义               |
| :---------- | :--- | :--- | :--- | :--- | :--------------------- |
| 低 (Low)    | 0    | 0    | 10   | 20   | 无明显热异常           |
| 中 (Medium) | 10   | 20   | 35   | 50   | 存在可疑热源           |
| 高 (High)   | 35   | 50   | 100  | 100  | 强烈热辐射（火灾特征） |

#### 2.1.2 气体风险 `TVOC`（单位：ppb）

| 模糊集      | a    | b    | c    | d    | 物理含义                         |
| :---------- | :--- | :--- | :--- | :--- | :------------------------------- |
| 低 (Low)    | 0    | 0    | 150  | 300  | 空气质量正常                     |
| 中 (Medium) | 150  | 300  | 700  | 1200 | 存在挥发物（可能为火灾初期热解） |
| 高 (High)   | 700  | 1200 | 5000 | 5000 | 大量 VOC 释放（强烈火灾征兆）    |

#### 2.1.3 湿度变化率 `RH_rate`（单位：%/帧，窗口=5帧线性回归斜率）

| 模糊集        | a    | b    | c    | d    | 物理含义                     |
| :------------ | :--- | :--- | :--- | :--- | :--------------------------- |
| 下降 (Down)   | -10  | -10  | -1.0 | -0.2 | 湿度快速下降（高温蒸发）     |
| 稳定 (Stable) | -0.5 | -0.1 | 0.1  | 0.5  | 湿度正常波动                 |
| 上升 (Up)     | 0.2  | 1.0  | 10   | 10   | 湿度快速上升（燃烧产水蒸气） |

#### 2.1.4 温度上升速率 `T_rise_rate`（单位：℃/帧，作为辅助修正因子）

| 模糊集          | a    | b    | c    | d    | 物理含义                 |
| :-------------- | :--- | :--- | :--- | :--- | :----------------------- |
| 缓慢 (Slow)     | -10  | -0.1 | 0.1  | 0.3  | 温度稳定                 |
| 中等 (Moderate) | 0.1  | 0.3  | 0.8  | 1.5  | 缓慢升温                 |
| 快速 (Fast)     | 0.8  | 1.5  | 10   | 10   | 急剧升温（火灾典型特征） |

> **注**：温度上升速率不直接作为规则输入，而是用于 **动态修正** 温度风险隶属度（即快速升温时，温度风险的高隶属度被增强）。

### 2.2 模糊规则库（Mamdani 型）

采用 **15 条核心规则**，覆盖温度、气体、湿度三者的典型组合场景。规则形式：`IF 温度风险 IS X AND 气体风险 IS Y [AND 湿度变化 IS Z] THEN 火灾风险等级 IS L`

| 规则号 | 温度风险 | 气体风险 | 湿度变化   | 湿度处理 | → 风险等级 |
| :----- | :------- | :------- | :--------- | :------- | :--------- |
| 1      | 低       | 低       | 任意       | 忽略     | **安全**   |
| 2      | 低       | 中       | 下降或稳定 | AND      | **关注**   |
| 3      | 低       | 中       | 上升       | AND      | **关注**   |
| 4      | 低       | 高       | 下降       | AND      | **关注**   |
| 5      | 低       | 高       | 稳定或上升 | AND      | **预警**   |
| 6      | 中       | 低       | 下降       | AND      | **关注**   |
| 7      | 中       | 低       | 稳定或上升 | AND      | **关注**   |
| 8      | 中       | 中       | 任意       | 忽略     | **预警**   |
| 9      | 中       | 高       | 下降或稳定 | AND      | **预警**   |
| 10     | 中       | 高       | 上升       | AND      | **报警**   |
| 11     | 高       | 低       | 下降       | AND      | **预警**   |
| 12     | 高       | 低       | 稳定或上升 | AND      | **预警**   |
| 13     | 高       | 中       | 下降       | AND      | **预警**   |
| 14     | 高       | 中       | 稳定或上升 | AND      | **报警**   |
| 15     | 高       | 高       | 任意       | 忽略     | **报警**   |

**规则设计原则**：

- **温度** 是最重要的单一证据——单独高温（`ΔT` 高）即可触发预警/报警。
- **气体** 是早期征兆——在温度尚未明显升高时，高 TVOC 可提前发出关注/预警。
- **湿度** 是辅助修正——极端湿度变化（急剧下降或上升）会强化已有判断，但不能单独触发高等级告警。
- **不一致处理**：当温度低但气体高时（规则 4、5），仅触发关注/预警而非报警，因为可能存在非火灾的 VOC 源（如清洁剂挥发）。

### 2.3 推理与去模糊化

**推理方法**：Mamdani 最小-最大合成（取小运算 `min` 作为规则强度，取大运算 `max` 合并相同输出类别的规则）。

**去模糊化**：重心法（质心法，Center of Gravity），计算输出概率的加权平均值。

**输出模糊集（火灾概率，0~100%）**：

| 等级           | 中心值 | 范围   | 说明                   |
| :------------- | :----- | :----- | :--------------------- |
| 安全 (Safe)    | 10     | 0~20   | 无风险                 |
| 关注 (Watch)   | 32.5   | 20~45  | 需留意                 |
| 预警 (Warning) | 60     | 45~75  | 可能火灾，建议人工确认 |
| 报警 (Alarm)   | 87.5   | 75~100 | 确认火灾，立即响应     |

**模糊逻辑输出为三个量**：

- `μ_Fire`：属于“火灾”的隶属度（0~1）
- `μ_NoFire`：属于“无火灾”的隶属度（0~1）
- `μ_Uncertain`：1 - μ_Fire - μ_NoFire（表示证据不确定性）

这三个量将作为 **D-S 证据理论的 BPA 输入**。

## 第三步：决策层融合 —— D-S 证据理论

D-S证据理论概念：

如果说“模糊逻辑”是把传感器的数值翻译成人类能懂的“语言”（温度高、气体中），那么 **“D-S 证据理论”就是一位“审判长”，负责综合多位证人的证词，在允许证人“不确定”或“撒谎”的情况下，做出最终判决。**

如果一个传感器认为火灾概率很大，一个认为概率小，D-S算法不会说“是发生火灾”，“不是发生火灾”，而是说明这次的数据结果不确定性很大，此时输出的结论是：`m(Fire)=40%`，`m(NoFire)=40%`，`m(Uncertain)=20%`。系统会告诉你：“我搞不清了，两个证人在打架，我建议你等下一帧数据再判断。”

D-S 证据理论（Dempster-Shafer Theory）是处理不确定性推理的经典方法，非常适合多传感器火灾探测。已有大量研究将其应用于多传感器火灾探测，并证明其能有效提高检测概率、降低误报。**强光干扰、传感器漂移、或者有人突然喷香水**都可能导致单一传感器误报

- **如果没有 D-S 理论（只有模糊逻辑）**：假设温度超标 20%，气体超标 80%，简单加权可能算出火灾概率 60%，直接触发报警，导致误报。
- **如果有 D-S 理论**：模糊逻辑会将“温度”和“气体”的数据转化为**两个独立的 Mass 函数**。D-S 理论发现“高温”和“高气体”虽然都支持火灾，但如果湿度数据**剧烈下降**（这与火灾燃烧时的剧烈蒸发冲突），D-S 理论会计算出高冲突系数，将最终结论判定为 **“高度不确定”**，从而**抑制误报警**，直到下一个周期数据稳定。

### 3.1 识别框架

定义识别框架 `Θ = {Fire, NoFire}`，所有可能的命题都是 `Θ` 的子集。

### 3.2 基本概率分配（BPA）

每个传感器（或特征）定义一个 **基本概率分配（BPA / mass 函数）**，满足：

```
m(∅) = 0
m(Fire) + m(NoFire) + m(Θ) = 1
```

其中 `m(Θ)` 表示不确定性（证据不支持任何具体假设）。

**三个证据源的 BPA 生成方式**：

| 证据源       | BPA 生成逻辑                                                 | 说明                                   |
| :----------- | :----------------------------------------------------------- | :------------------------------------- |
| **温度证据** | 由模糊逻辑输出的 `μ_Fire_T`、`μ_NoFire_T`、`μ_Uncertain_T` 直接映射 | 温度是最直接的火灾证据，权重最高       |
| **气体证据** | 由模糊逻辑输出的 `μ_Fire_G`、`μ_NoFire_G`、`μ_Uncertain_G` 直接映射 | 气体是早期征兆，但在通风条件下可能稀释 |
| **湿度证据** | 由模糊逻辑输出的 `μ_Fire_H`、`μ_NoFire_H`、`μ_Uncertain_H` 直接映射 | 湿度是辅助证据，不确定性较高           |

### 3.3 D-S 组合规则

对于两个 BPA `m₁` 和 `m₂`，Dempster 组合规则定义如下：

**冲突系数**（反映证据间的矛盾程度）：

```
K = m₁(Fire) × m₂(NoFire) + m₁(NoFire) × m₂(Fire)
```

**组合后的 mass 函数**（`K < 1` 时）：

```
m₁₂(Fire)   = [m₁(Fire)·m₂(Fire) + m₁(Fire)·m₂(Θ) + m₁(Θ)·m₂(Fire)] / (1 - K)
m₁₂(NoFire) = [m₁(NoFire)·m₂(NoFire) + m₁(NoFire)·m₂(Θ) + m₁(Θ)·m₂(NoFire)] / (1 - K)
m₁₂(Θ)      = [m₁(Θ)·m₂(Θ)] / (1 - K)
```

**完全冲突处理**（`K ≥ 0.999` 时）：认为传感器失效，输出 `m(Fire)=0, m(NoFire)=0, m(Θ)=1`（完全不确定），等待下一帧数据。

**多证据组合**：三个证据源按顺序两两组合——先组合温度和气体，再将结果与湿度组合。

### 3.4 决策判据

根据组合后的最终 BPA `{m(Fire), m(NoFire), m(Θ)}` 做出决策：

| 决策结果           | 触发条件                               | 说明                 |
| :----------------- | :------------------------------------- | :------------------- |
| **报警 (Alarm)**   | `m(Fire) ≥ 0.75` 且 `m(Θ) ≤ 0.25`      | 高置信度火灾         |
| **预警 (Warning)** | `m(Fire) ≥ 0.55` 且 `m(Θ) ≤ 0.35`      | 中等置信度，建议确认 |
| **关注 (Watch)**   | `m(Fire) ≥ 0.35` 且 `m(NoFire) < 0.50` | 低置信度，持续观察   |
| **安全 (Safe)**    | `m(NoFire) ≥ 0.60` 或 `m(Θ) > 0.60`    | 无火灾证据或证据不足 |

## 第四步：多级报警机制与系统逻辑

### 4.1 多级报警机制

| 等级               | 触发条件                               | 系统响应                                                 |
| :----------------- | :------------------------------------- | :------------------------------------------------------- |
| **安全 (Safe)**    | `m(Fire) < 0.35`                       | 仅记录数据，无动作                                       |
| **关注 (Watch)**   | `m(Fire) ≥ 0.35` 且 `m(NoFire) < 0.50` | 记录数据，日志标记，MQTT 上报状态                        |
| **预警 (Warning)** | `m(Fire) ≥ 0.55` 且 `m(Θ) ≤ 0.35`      | 发送 MQTT 预警消息，触发本地蜂鸣器（短促提示）           |
| **报警 (Alarm)**   | `m(Fire) ≥ 0.75` 且 `m(Θ) ≤ 0.25`      | 发送 MQTT 紧急告警，启动本地声光报警，指示牌切换逃生方向 |

### 4.2 时间序列稳定性增强

为防止瞬时波动导致误报，引入以下滤波和确认机制：

**逐特征滤波**：

- 对 `ΔT`、`TVOC`、`RH` 应用 **移动平均滤波**（窗口大小 = 3~5 帧），平滑噪声。
- 对 `T_rise_rate` 和 `RH_rate` 本身已通过线性回归计算，具有天然的平滑效果。

**报警确认机制**：

- 从“关注”升级到“预警”需要 **连续 3 帧** 满足条件，防止尖峰噪声。
- 从“预警”升级到“报警”需要 **连续 2 帧** 满足条件，平衡响应速度与误报率。

**降级与恢复**：

- 若当前处于报警/预警状态，连续 5 帧满足“安全”条件则降级，避免因短暂传感器波动导致告警震荡。

### 4.3 与现有系统的集成

| 输出           | 目标                     | 协议/格式                      |
| :------------- | :----------------------- | :----------------------------- |
| 火灾概率、等级 | MQTT Broker → Go 后端    | JSON，主题 `fire_alert`        |
| 声光报警       | 本地 ESP32 外设          | GPIO 控制蜂鸣器、LED           |
| 指示牌控制     | MQTT Broker → 指示牌节点 | JSON，主题 `signboard/control` |
| 数据存储       | InfluxDB（时序数据库）   | 通过 MQTT → Go 后端写入        |

## 五、参数汇总与调优指南

### 5.1 隶属度函数阈值汇总

| 参数         | 符号          | 低→中→高阈值           | 说明                                     |
| :----------- | :------------ | :--------------------- | :--------------------------------------- |
| 温度温差     | `ΔT`          | 20 / 35 / 50 °C        | 根据室内/室外环境调整，商场建议 15/30/45 |
| TVOC         | `TVOC`        | 300 / 700 / 1200 ppb   | 根据建筑类型调整，办公室较低，工厂较高   |
| 湿度变化率   | `RH_rate`     | -1.0 / -0.1 / 0.5 %/帧 | 取决于通风条件，密闭空间变化更敏感       |
| 温度上升速率 | `T_rise_rate` | 0.3 / 0.8 / 1.5 °C/帧  | 影响温度证据的权重                       |

### 5.2 决策阈值汇总

| 决策阈值            | 推荐值 | 高灵敏度 | 高可靠性 | 说明                |
| :------------------ | :----- | :------- | :------- | :------------------ |
| 报警触发 `m(Fire)`  | ≥ 0.75 | ≥ 0.65   | ≥ 0.85   | 降低误报 / 降低漏报 |
| 预警触发 `m(Fire)`  | ≥ 0.55 | ≥ 0.45   | ≥ 0.65   | 同上                |
| 最大不确定度 `m(Θ)` | ≤ 0.25 | ≤ 0.35   | ≤ 0.15   | 过高时暂不决策      |
| 报警确认帧数        | 2 帧   | 1 帧     | 3 帧     | 响应速度 vs 稳定性  |
| 降级确认帧数        | 5 帧   | 3 帧     | 8 帧     | 防止告警震荡        |

### 5.3 不同场景的参数建议

| :---------------------- | :------- | :------------ | :----------- | :------------------------- |

## 六、计算量评估

以下评估在 ESP32（240MHz）平台上进行理论估算，**完全不依赖具体代码实现**：

| 计算阶段         | 主要操作                                            | 估算浮点运算量       | 估算耗时（240MHz） |
| :--------------- | :-------------------------------------------------- | :------------------- | :----------------- |
| **特征提取**     | 线性回归（窗口5点）、统计最大值、面积计算           | ≈ 100 次乘加         | < 10 μs            |
| **模糊逻辑**     | 梯形隶属度（3×3=9次）、规则推理（15条取小）、重心法 | ≈ 150 次浮点运算     | < 15 μs            |
| **D-S 组合**     | 2 轮组合，每轮 3×3 乘加                             | ≈ 40 次浮点运算      | < 5 μs             |
| **总计（每帧）** |                                                     | **≈ 300 次浮点运算** | **< 50 μs**        |

> 整帧计算耗时 < 50 微秒，即使按 4Hz 刷新率（250ms/帧），CPU 占用率 < 0.02%。该算法在绝大多数主流微控制器上均可实时运行，无需依赖服务器计算。

## 七、总结

1. **数据层**：利用 AHT2x 的温湿度数据对 MLX90640 和 ENS160 进行补偿。
2. **特征层**：提取温度（最高温、温差、升温速率）、气体（TVOC）、湿度变化率等关键特征。
3. **特征层融合**：采用 **模糊逻辑** 将特征值映射为三个证据源的 BPA，处理传感器数据的不确定性和非线性关系。
4. **决策层融合**：采用 **D-S 证据理论** 组合三个证据源，显式处理证据冲突和不确定性，获得比单一传感器更可靠的火灾概率判断。
5. **多级报警**：安全 → 关注 → 预警 → 报警，结合时间序列确认机制，降低误报率。
6. **计算轻量**：整帧计算 < 50 μs，可在 ESP32 等微控制器上本地实时运行，无需依赖云端算力。

这套“由下至上”的融合架构，能让您的系统不仅“看得见”温度，还能“闻得到”气味，从而实现更智能、更可靠的火灾预警。
